# 架構憲法 — 5 條鐵律 + Domain Boundaries
## Architectural Constitution v1.0

**祺驊 CHI HUA · 供應鏈作戰系統**

> 這是系統的**不可違反原則**（不是建議、是憲法）。
> 任何新功能 / 新模組 / 新整合都必須遵守。
> 違反 = 系統爛掉、無法 scale、AI 互相矛盾、信任崩塌。
>
> Code Review / PR 都必須驗證這 5 條。違反就 BLOCK，不容妥協。

---

## 鐵律 1：所有功能都必須 Event-Driven

### 重要性：**極為重要**

### 禁止
- ❌ 頁面直接呼叫
- ❌ Page A 直接改 Page B 的資料
- ❌ 繞過 Event Stream 直接修改狀態

### 必須
- ✅ `Event → Workflow → Decision → Action`
- ✅ 所有狀態變化 emit 事件
- ✅ 下游模組訂閱 Event Stream 接收

### 範例
```
❌ 錯誤：ASN page → 直接更新 WO page
✅ 正確：asn_delayed event → workflow engine → wo risk recalculated
```

### 為什麼重要
**Event Architecture 才能 scale** — 否則 N 個頁面需要 N² 個耦合，永遠改不動。

### 違反的後果
改一個地方、別處還是舊資料 → 決策不一致、靠人工同步 → 出包。

---

## 鐵律 2：UI 不准有 Business Logic

### 重要性：**超重要**

### 禁止
- ❌ React page 寫算法
- ❌ 頁面 component 內做 if/else 業務判斷
- ❌ 前端複寫 Engine 已有的計算

### 必須
- ✅ AI Engine 統一計算
- ✅ UI 只能 **Surface**（呈現）
- ✅ 所有業務規則收斂在 `lib/erp/engine-*.ts`

### 範例
```
❌ 錯誤：在 page.tsx 內計算 OTD = onTime / total
✅ 正確：lib/erp/otif.ts computeOTD() → page 只呼叫
```

### 為什麼重要
- 每頁算法不同
- AI 不一致
- KPI 失真
- 最後全錯

### 違反的後果
UI 改版一次、業務規則就掉一塊 → 永遠 debug。

---

## 鐵律 3：所有 AI 必須 Centralized

### 重要性：**該章節要重要**

### 禁止
- ❌ 每個部門做自己的 AI
- ❌ 採購做一套 ETA、業務做另一套 ETA
- ❌ 10 個頁面 10 套預測模型

### 必須
- ✅ **AI Core Layer 統一**
- ✅ `eta-engine`（單一 ETA 來源）
- ✅ `risk-engine`（單一風險評分）
- ✅ `cost-engine`（單一成本拆解）
- ✅ `simulation-engine`（單一情境模擬）

| Engine | 功能 |
|---|---|
| eta-engine | ETA |
| risk-engine | 風險 |
| cost-engine | 成本 |
| simulation-engine | 模擬 |

### 範例
```
❌ 錯誤：supplier-portal 算一個 ETA，eta-forecast 算另一個 ETA
✅ 正確：兩頁都呼叫 lib/erp/eta-forecast.ts 的同一個函式
```

### 為什麼重要
**AI 會互相矛盾** → 老闆看到兩套不同數字 → 不再信任系統。

### 違反的後果
AI 訊號變雜訊 → 重要警報被忽略 → 系統失效。

---

## 鐵律 4：所有資料都必須有 Source of Truth

### 重要性：**絕不能模糊化**

### 禁止
- ❌ 同一筆 ETA 存兩個地方
- ❌ Portal 一份、ERP 一份，誰真誰假不知道
- ❌ 本系統重複造一份鼎新已有的資料

### 必須
- ✅ **Single Source of Truth（SSoT）** — 每個 Entity 只有一個

| Data | Source |
|------|--------|
| PO | ERP（鼎新唯讀）|
| ASN | Portal（本系統）|
| ETA | ETA Engine（本系統計算）|
| Risk Score | Risk Engine（本系統計算）|
| Inventory | ERP（唯讀，扣帳回 ERP）|

### 範例
```
❌ 錯誤：同一個 ETA 在 supplier-portal、eta-forecast、戰情室各存一份
✅ 正確：ETA 只在 eta-engine 計算，其他模組透過 API 取
```

### 為什麼重要
**不然 AI 會開始亂掉** — 用哪份資料算的？答案不一樣。

### 違反的後果
資料漂移、衝突、人工協調成本爆炸。

---

## 鐵律 5：所有 Action 必須可追溯

### 重要性：**信任的基礎**

### 禁止
- ❌ AI 黑箱判定（不知道為什麼）
- ❌ 人工偷偷修改數據（無 audit）
- ❌ Workflow 跑了沒人知道

### 必須
- ✅ **Audit Trail 全程記錄**

| 動作類型 | 必須記錄 |
|---|---|
| AI 訊號 | ✅ Inputs / Logic / Confidence |
| 人工修改 | ✅ 記名 + 原因 + 時間 |
| Workflow | ✅ trace |
| ETA 變更 | ✅ 留前後對照 |
| Cost 變更 | ✅ 留審批人 |

### 範例
```
❌ 錯誤：AI 突然把供應商 A 降到 D 級，沒人知道為什麼
✅ 正確：Audit log 顯示「基於 X/Y/Z 3 個事件 + risk-engine v2 規則 → 自動降級，可重放」
```

### 為什麼重要
**企業永遠不能 fully trust AI**（除非可追溯）。

### 違反的後果
AI 變黑箱、合規檢查過不了、出事沒人擔。

## 🗺 Domain Boundaries（領域邊界）

### 5 個 Domain

| Domain | 負責 | 擁有 Entity | 對外 Emit | 訂閱 |
|---|---|---|---|---|
| **🤝 Supplier Domain** | 供應商互動 | Supplier / PO / ASN / Quality | po_unconfirmed / asn_missing / asn_delayed / iqc_ng | raw_material_surge |
| **🏭 Manufacturing Domain** | 工單 / 生產 | Work Order / Equipment / BOM | eta_changed / stockout_imminent / delay_signal | asn_missing / iqc_ng |
| **📦 Inventory Domain** | 庫存 / 倉儲 | Stock / Storage / Receiving | receiving_fail_hold / stockout_imminent | asn_delayed / po_unconfirmed |
| **🧠 AI Domain** | 預測 / 推薦 | Prediction / Twin / Recommendation | supplier_score_drop / raw_material_surge | **ALL events**（全域訂閱者） |
| **🔐 Governance Domain** | 權限 / 主檔 / 審計 | User / Role / MDM / Audit Log | （只記錄不發業務事件） | **ALL events**（用於 audit）|

### 核心原則

> **Domain 之間：只能透過 Event 溝通**

### 不允許
- ❌ Domain A 的程式碼直接讀 Domain B 的資料庫
- ❌ Domain A 的 UI 直接呼叫 Domain B 的內部 function
- ❌ Domain 之間共用 mutable state

### 必須
- ✅ A 完成事情 → emit event
- ✅ B 訂閱 event → 自己處理
- ✅ 任何跨 domain 通訊 = 一條清楚的事件路徑（可 trace）

### 為什麼
- 各 domain 可獨立部署、獨立測試、獨立 scale
- 改一個 domain 不會炸到別人
- Audit Trail 自動成形

## ⚖️ 違反憲法的後果總結

| 違反 | 後果 |
|---|---|
| #1 Event-Driven | N² 耦合，永遠改不動 |

## 📜 程式碼審查 (Code Review) Checklist

任何 PR 進入 main 前，**必須通過以下檢查**：

- [ ] **鐵律 1**：新功能是否有 event emit？跨模組通訊是否走 Event Stream？
- [ ] **鐵律 2**：UI 是否只有呈現邏輯？業務規則是否在 `lib/erp/engine-*.ts`？
- [ ] **鐵律 3**：是否引入新 AI/預測函式？該函式是否在 Core Layer 唯一？
- [ ] **鐵律 4**：新 Entity 是否在 MDM Registry 註冊？SoT 是否定義清楚？
- [ ] **鐵律 5**：所有變更動作是否有 audit log？AI 判定是否可解釋？
- [ ] **Domain 邊界**：跨 domain 呼叫是否走 Event？沒有直接資料庫讀寫？

任何一項 ✗ → **直接 BLOCK PR，不能 merge。**
